# Пояснительная записка к проекту QuizRoom

## 1. Назначение проекта

QuizRoom — прототип веб-приложения для проведения интерактивных квизов. Организатор создаёт квиз, добавляет текстовые вопросы или вопросы с изображениями, настраивает время и запускает игровую комнату. Участники проходят авторизацию или входят гостями по QR-коду, отвечают на текущий вопрос и получают пояснение после раунда. После каждого вопроса отображаются рейтинг и статистика, а после завершения — победитель и итоговый лидерборд. Результаты сохраняются в личных кабинетах.

## 2. Выбор средств разработки

Для клиентской части выбран React, поскольку компонентный подход удобен для реализации нескольких состояний интерфейса: лобби, активного вопроса, ожидания, редактора и лидерборда. Vite обеспечивает быстрый запуск проекта и компактную производственную сборку.

Сервер реализован на Node.js с Express. Это позволяет использовать JavaScript на клиенте и сервере и упрощает учебную разработку. Для обмена данными во время игры применяется Socket.IO. В отличие от периодических HTTP-запросов, постоянное соединение позволяет одновременно отправить вопрос всем участникам, обновлять количество ответивших и показывать результаты без перезагрузки страницы.

В качестве базы данных выбрана SQLite. Она не требует отдельного сервера, хранится в одном файле и подходит для демонстрационного MVP. Пароли хешируются bcrypt, а авторизация выполняется с помощью JWT. Изображения сохраняются в локальном каталоге приложения.

## 3. Архитектура и модель данных

Приложение состоит из React-клиента, REST API, игрового Socket.IO-сервера и базы SQLite. REST API используется для регистрации, авторизации, редактирования квизов, загрузки изображений и получения истории. События Socket.IO обслуживают текущую игровую сессию.

Основные таблицы базы данных: пользователи, квизы, вопросы, игровые сессии, участники сессий и ответы. Связь пользователя с квизом определяет владельца. Каждая игровая сессия относится к одному квизу и имеет уникальный код комнаты. Для каждой пары «участник — вопрос — сессия» допускается только один ответ, что исключает повторное начисление баллов.

## 4. Реализованные этапы

1. Определены пользовательские сценарии организатора и участника.
2. Спроектирована реляционная модель данных и ограничения целостности.
3. Разработаны адаптивные экраны авторизации, кабинета, редактора, подключения, игры и результатов.
4. Реализованы REST API, JWT-авторизация и разграничение ролей.
5. Настроены Socket.IO-комнаты, единый серверный таймер, приём ответов и начисление баллов.
6. Добавлены история, демонстрационные пользователи и тестовый квиз.
7. Выполнены API-тесты, полный автоматический сценарий игры и визуальная проверка интерфейса.
8. Добавлены QR-код комнаты, гостевой вход, восстановление после обновления, пояснения, аналитика и отключаемые звуки.

## 5. Подсчёт баллов и тестирование

Правильность ответа проверяется на сервере: для множественного выбора множества выбранных и правильных вариантов должны полностью совпадать. За правильный ответ начисляется от 60 до 100 процентов базовой стоимости вопроса. Итог зависит от оставшегося времени, поэтому быстрый ответ получает больше баллов. Ошибочный ответ даёт ноль баллов.

Пять автоматических API-тестов проверяют доступность сервера, авторизацию, создание квиза и вопроса с пояснением, гостевой вход и разграничение ролей. Отдельный Socket.IO smoke-test создаёт комнату, подключает участника, запускает вопрос, отправляет ответ, восстанавливает соединение и проверяет итоговый лидерборд со статистикой. Дополнительно вручную проверены QR-код, объяснение ответа, аналитика и восстановление комнаты после перезагрузки.
